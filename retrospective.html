Workflow

  • Dev branches from Master and each team member uses their own development branch for feature specific branches
  • Our team used various techniques and strategies in order to distribute tasks effictivly. We mainly used, githubs issues and the meetings in order to distribute tasks.
  • Framworks
  • Back-end: Grails, rest, groovy, java, hibernate.
  • Front-end: react, html, css, javascript, google API (for calendar and authentication).

MVC/Design Patterns

  • Everything was dependent on Rest.
  • Server-side REST APIs was implemented
  • Model: Grails domain classes
    View: Html, Javascript, css
    Controller: Grails controller
  • Communication is throught the controllers
    Eg: User logs in (to front end), data sent to controller method, controller method sends to appropriate model class, data is verified by the model (our database) and then sent back to the controller which sends it to the appropriate view.
  • Improvments
  • More comments, understand all the technology we were going to use beforehand, and consistency through out the application.

Project Retrospective

  • Project Strengths
  • We communicated well between team members, we handed almost everything in on time, we generally used our frameworks and technologies properly, and we had high attendance at all meetings. Overall, we think we did pretty well on the project.
  • What to improve
  • UI design, integration.
  • We liked how Grails had everything in one place, everything fit together. It was simple once we learned it. Domain classes were also simple. CSS, HTML, and Javascript were relatively simple.
  • We didn't like Grails' guide and resources had limited useful information making it difficult to learn. There were many conflicting ways to do a single task. Grails was also difficult to setup, and had long load times. Fetch concept was a lot of work, having to go to backend every time we needed something.
