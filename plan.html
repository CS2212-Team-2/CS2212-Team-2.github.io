Title of New Feature: Events
 Description
 The events function allows users to add events to their Gmail calendar. The user can add others (roommates/siblings/parents/children) to the event. The restriction to attendees is the person must be a member of the same group (house). The scheduled event will populate the creator and attendees to their respective calendars. Users will view their Gmail calendar from our program. Feasibility
The addition of an events function is feasible. A successful implementation will require use of the google Gmail API. The function will require the events creator to make a secure connection to his/her calendar and the calendar of others. To determine feasibility, we have successfully tested this functionality using: https://developers.google.com/apis-explorer/#s/calendar/v3/
Dependancy
The events function is dependent on the Task class and the google calendar API. If the calendar function is unavailable, the program should work uninterrupted.

Title of New Feature: Leaderboard
 Description
 The leaderboard tracks each house members’ performance at completing task and meeting financial obligations. The user will receive points if tasks and financial commitments are completed by due dates. The user will lose points for non-compliance. The user will receive a grade based on the following equation:
Grade = (Points Earned)/(Total Points Available)
Bonus/reward can be award to the leader at the end of each increment. Feasibility
The implementation of the leaderboard is feasible. A successful implementation will require the proper tracking and storing the user “Grade”. We anticipate using a simple database to store and update the “Grade.”
Dependancy
The leader board function is dependent on the information feed to from the tasks and financial classes. If either class fails, then the leaderboard will not work. If the leaderboard function fails, its failure will not have a systemic impact on the proper functioning of the program.

Title of New Feature: User analysis and recommendation
Description
 An advance addition building on the leaderboard feature, where the program analyzes how the users grade patterns. If any patterns are found in how they are losing points, then we can provide recommendations or personal notifications to the user on how they can improve. Feasibility
The implementation of the leaderboard is feasible. A successful implementation will require to compare the resulting grade of an user for each week/month. Based on a threshold we can determine if the user is losing out on potential points because of failing a certain responsibility.
E.g. Threshold 12 weeks and failure rate >75%
Out of the past 12 weeks the user has failed to take-out the trash in 9 of the weeks resulting in a poor leaderboard placement.
Dependancy
This feature is dependent on the leaderboard as well as how it is implemented. However no other parts of the application will depend on this feature.

Title of New Feature: Groceries, Toiletries and Misc Count
Description
 This feature can keep a detailed list of all the “things” that housemates share and might need to buy on a weekly/monthly basis. Such as toiletries, some types of groceries, etc. The feature will also have an empty, low, full tag attached to each “thing”. E.g toilet paper “empty” (lol) Feasibility
Implementation of this seems very simple. The user can enter what “things” they would like on this list and all the users can update the status tag. The class would have an object name “user enters” and a status tag.
Dependancy
This feature is independent of any other features and no other parts of the application will depend on this feature.
